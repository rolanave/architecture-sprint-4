# Задание 2. Мониторинг

## 1. Мотивация

Уже сейчас клиенты жалуются на задержки в выполнении заказов и задержках при выполнении запросов,
что приводит отрицательному опыту и в итоге к оттоку клиентов (плохие отзывы, потери контрактов).
В итоге компания может начать терять деньги (отказы клиентов, суды и прочее) при недозагруженном производстве.

Это является следствием того, что. внутренняя работа решения сейчас непрозрачна, что приводит к тому что
1. Поддержка начинает решать проблему реактивно (после обращения клиентов). 
2. Решение проблемы занимает больше времени. Как результат необходимо выделять больше ресурсов (человек) на поддержку продукта и клиентов.

Также расширение текущего решения будет более дорогим, так как невозможно понять узкие места системы.

Внедрение мониторинга позволит повысит прозрачность системы, что позволит
- поддержке проактивно решать возникающие проблемы
- снизит стоимость поддержки (изучение проблемы будет происходить быстрее), что позволит сократить команду поддержки, а улучшит опыт пользователей
- понять как эффективнее использовать текущую систему, а значит снизит стоимость расширения системы
  (так как позволит определять и фокусироваться на расширении узких мест) системы

## 2. Выбор подхода к мониторингу

Предлагается комбинированный подход к мониторингу:

LTES (latency, traffic, errors, saturation) - для веб приложений ( MES, CRM, Internet shop):
	latency - время отклика на действие пользователя
	traffic - количество запросов на данную функцию
	errors - количество ошибок, которое возвращается пользователю
	saturation - сколько памяти приложение занимает в пользовательском окружении

RED (rate, errors, duration) - для микросервисов (MES-API, CRM-API, Shop-API)
	rate - сколько запросов в единицу времени приходит от пользователей
	errors - сколько запросов оканчивается успехом, а сколько ошибками (включая тайм-ауты)
	duration - сколько по времени занимает один запрос 

USE (utilizaition, saturation, errors) - для микросервисов (MES-API, CRM-API, Shop-API) и баз данных
	utilization - сколько микросервис занимает процессорной мощности 
	saturation -  сколько запросов стоит в очереди, сколько запросов отвергается изза переполнения очереди (RabbitMQ для MES-API, CRM-API)
	errors - количество ошибок внутри микросервисов (например ошибки доступа к базе данных)

1. Определить список метрик для сбора и список алертов и трешхолдов на базе метрик
2. Развернуть систему сбора метрик (например прометеус)  и настроить ее
	- настроить откуда прометеус будет забирать метрики
	- развернуть Time-Series базу данных 
	- настроить алерты для настроенных метрик
3. Настроить систему визуализации настроенных метрик (например графану)
